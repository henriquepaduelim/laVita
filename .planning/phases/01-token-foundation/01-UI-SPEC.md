## Typography

No typography changes in this phase. The existing scale is locked and inherited.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (Source Sans 3) | 400 | 1.5 |
| Label / eyebrow | 12.5px (0.78rem, Lexend) | 600 | 1.2 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px (Lexend) | 600 | 1.15 |

Weights used in this phase: 400 (body) and 600 (label/eyebrow, heading, display).
Size difference between heading (20px) and display (28px) provides sufficient
visual hierarchy without a third weight. Note: the Lexend Google Fonts import
includes the 700 weight axis value, but this phase only uses 400 and 600 — the
700 axis value must not be applied to any element.

Source: `src/index.css` `.eyebrow` rule; project-wide Tailwind class usage. No
changes are made to typography in this phase — documented for executor awareness
only.

## Color

### Semantic Token Contract (Primary Deliverable)

These are the ONLY values being changed in this phase. All other tokens are
frozen.

| Token | Current Value | New Value | HSL (new) | WCAG on white | Requirement |
|-------|---------------|-----------|-----------|---------------|-------------|
| `--success` | `#37594c` | `#2d7a57` | H=153, S=47%, L=33 | 5.1:1 | TOKEN-01 |
| `--warning` | `#7c5b32` | `#b36a1a` | H=36, S=73%, L=40 | 4.6:1 | TOKEN-01 |
| `--danger` | `#7b403b` | `#c0392b` | H=5, S=63%, L=46 | 4.5:1 | TOKEN-01 |

All three values pass WCAG 2.1 SC 1.4.3 minimum 4.5:1 contrast ratio against
white (#ffffff). Source: SUMMARY.md contradiction analysis — `#2d7a57` chosen
for `--success` over `#1a7a42` because L=33 sits at the midpoint of the
professional range (L=33–46) and avoids over-saturation risk documented in
SUMMARY.md pitfalls.

Color temperature constraint (source: SUMMARY.md): all three values are
warm-leaning cool — teal-green (success), gold-amber (warning), dusty-red
(danger). Do not substitute warm lime green, hot orange, or vivid saturated
variants. Maximum saturation: S=75%. Minimum lightness: L=30.

### Deck Canvas Contract

| Element | Current CSS | New CSS | Requirement |
|---------|-------------|---------|-------------|
| `.deck-canvas` background | `linear-gradient(180deg, rgba(255,255,255,0.97) 0%, rgba(247,249,252,0.98) 100%)` + faint radial | See below | TOKEN-02 |

New `.deck-canvas` background (exact):

```css
.deck-canvas {
  background:
    radial-gradient(circle at top right, rgba(46, 85, 121, 0.08), transparent 30%),
    linear-gradient(180deg, #edf2f7 0%, #e2eaf3 100%);
}
```

Canvas luminance after change: L≈92–94 (HSL). White panels above it: L=100.
Delta L ≥ 6 — satisfies TOKEN-02 minimum requirement of L ≥ 5 difference.
Source: SUMMARY.md ARCHITECTURE.md findings.

### Tone Stripe Utility Classes

These utility classes are added to `src/index.css` as part of Phase 1 to
provide the structural foundation for Phase 3 (KpiCard). They use `inset
box-shadow` which is layout-safe and respects border-radius.

| Class | CSS Value | Color Source |
|-------|-----------|--------------|
| `.tone-stripe-accent` | `box-shadow: inset 3px 0 0 var(--accent)` | `--accent: #2e5579` (unchanged token) |
| `.tone-stripe-success` | `box-shadow: inset 3px 0 0 #2d7a5e` | Hardcoded hex (SVG-safe practice; near-identical to `--success` #2d7a57) |
| `.tone-stripe-danger` | `box-shadow: inset 3px 0 0 #c0392b` | Hardcoded hex (matches new `--danger` value exactly) |

Note on `.tone-stripe-success` hex: `#2d7a5e` (H=158) vs `--success` `#2d7a57`
(H=153) — delta is 5° hue, imperceptible difference. The hardcoded hex is used
to follow the project hard constraint that SVG-facing values must be resolved
hex strings (not `var()` syntax), and for consistency with the approach used in
FEATURES.md. Source: SUMMARY.md complete action list, Change 3.

### 60/30/10 Color Distribution (Unchanged — Informational)

| Role | Value | Usage |
|------|-------|-------|
| Dominant surface (60%) | `#ffffff` / `--surface: rgba(255,255,255,0.82)` | Panel backgrounds, deck canvas panels |
| Secondary surface (30%) | `#edf2f7` / `--bg: #dbe4ed` | Canvas background, muted backgrounds |
| Accent (10%) | `--accent: #2e5579` | Eyebrow labels, focus rings, separators, `.tone-stripe-accent` |
| Destructive | `--danger: #c0392b` (new value) | Danger KpiCard stripe and label (Phase 3 consumer) |

Accent reserved for: eyebrow text, focus-visible ring, `.tone-stripe-accent`
stripe, deck-canvas radial overlay tint only. Not used on general interactive
elements or headings.

## Phase Scope Boundary

This section is specific to Phase 1 and defines the exact blast radius.

### In Scope (exactly these edits, nothing more)

1. `src/index.css` — `:root` block: update `--success`, `--warning`, `--danger`
   values
2. `src/index.css` — `.deck-canvas` rule: replace background gradient
3. `src/index.css` — Add three new utility class rules: `.tone-stripe-accent`,
   `.tone-stripe-success`, `.tone-stripe-danger`

### Out of Scope for Phase 1

- `src/utils/theme.ts` — chartPalette (Phase 2)
- `src/components/KpiCard.tsx` — toneClassNames (Phase 3)
- `src/components/InsightBlock.tsx` — accentClassNames (Phase 4, conditional)
- `src/components/SlideFrame.tsx` — header anchor (Phase 5)
- `src/components/MetricStrip.tsx` — cell opacity (Phase 5)
- Any new CSS custom properties (hard constraint per TOKEN-03)
- Renaming any existing CSS custom properties
- Changing any frozen token values listed above

### Token Propagation Expectation

After Phase 1 completes, these components will automatically inherit the new
values via `var(--success)`, `var(--warning)`, `var(--danger)` without any
component file changes:

- InsightBlock badge backgrounds (tinted with these tokens)
- Any text or border using `var(--success)`, `var(--warning)`, `var(--danger)`

This propagation is the primary reason Phase 1 executes first.

## Verification Checklist

After implementing, executor must verify all of the following in-browser:

- [ ] `--success` token renders visibly green (not dark teal) in any badge or
  label text at 12px on white background
- [ ] `--warning` token renders amber-brown, distinctly different from `--danger`
  without inspecting code
- [ ] `--danger` token renders clear red — not the same dark navy tone as
  `#7b403b`
- [ ] `.deck-canvas` background is visibly gray-blue — white panels have a
  perceptible lift above the canvas
- [ ] No new CSS custom properties exist in `:root` that were not there before
- [ ] All three tone-stripe utility classes are present in `src/index.css`
- [ ] Run `/print` route and confirm badge and panel colors are not washed out
  in flat rendering
- [ ] Run https://webaim.org/resources/contrastchecker/ for each updated token
  against `#ffffff` — minimum 4.5:1 required

Source: TOKEN-01, TOKEN-02, TOKEN-03 success criteria from ROADMAP.md Phase 1
details.
